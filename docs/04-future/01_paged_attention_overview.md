## 1. 메모리 단편화란?

메모리 단편화(Memory Fragmentation)는 **사용 가능한 메모리가 충분함에도 불구하고, 효율적으로 활용하지 못하는 현상**을 말한다. 크게 **외부 단편화**와 **내부 단편화** 두 가지로 나뉜다.

### 1.1 외부 단편화 (External Fragmentation)

외부 단편화는 메모리의 빈 공간이 여기저기 흩어져 있어서, **총 빈 공간은 충분하지만 연속된 큰 공간을 확보할 수 없는 상태**를 뜻한다.

**비유: 주차장 문제**

```
[ 주차장 — 외부 단편화 예시 ]

┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 차 A │ 빈칸 │ 차 B │ 빈칸 │ 빈칸 │ 차 C │ 빈칸 │ 차 D │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
                                 ↑
                    빈칸 4개지만 연속 2칸이 최대
                    → 3칸짜리 대형버스 주차 불가!
```

주차장에 빈 칸이 총 4개 있지만, 연속된 빈 칸은 최대 2개뿐이다. 3칸이 필요한 대형버스는 주차할 수 없다. 이것이 외부 단편화다.

### 1.2 내부 단편화 (Internal Fragmentation)

내부 단편화는 **할당받은 메모리 공간 안에서 실제로 사용하지 않는 부분이 낭비되는 현상**이다.

**비유: 택배 상자 문제**

```
[ 택배 상자 — 내부 단편화 예시 ]

    ┌─────────────────────────┐
    │ ┌─────┐                 │
    │ │ 물건 │    빈 공간      │   ← 작은 물건에 큰 상자 사용
    │ │     │   (낭비!)       │      실제 사용: 20%
    │ └─────┘                 │      낭비: 80%
    └─────────────────────────┘
```

작은 물건을 보내는데 큰 상자밖에 없어서 큰 상자를 쓰면, 상자 안의 대부분이 빈 공간으로 낭비된다. 메모리에서도 마찬가지로, 100바이트만 필요한데 4KB 블록을 할당받으면 나머지 공간이 낭비된다.

### 1.3 두 단편화의 비교

```
┌──────────────┬────────────────────────┬────────────────────────┐
│     구분     │     외부 단편화        │     내부 단편화        │
├──────────────┼────────────────────────┼────────────────────────┤
│ 발생 위치    │ 할당된 블록 사이(외부) │ 할당된 블록 내부       │
│ 원인         │ 메모리 할당/해제 반복  │ 고정 크기 블록 할당    │
│ 결과         │ 연속 공간 확보 불가    │ 블록 내 미사용 공간    │
│ 해결 방법    │ 페이징, 압축(compaction)│ 블록 크기 최적화       │
└──────────────┴────────────────────────┴────────────────────────┘
```

## 2. OS의 페이징 기법

운영체제(OS)는 메모리 단편화 문제를 해결하기 위해 **가상 메모리(Virtual Memory)**와 **페이징(Paging)** 기법을 사용한다. PagedAttention의 핵심 영감이 바로 이 OS 기법에서 왔으므로, 먼저 이를 이해해야 한다.

### 2.1 가상 메모리의 기본 개념

프로그램은 실제 물리 메모리(RAM)의 위치를 직접 다루지 않는다. 대신 **가상 주소(Virtual Address)**를 사용하고, OS가 이를 **물리 주소(Physical Address)**로 변환해 준다.

```
[ 가상 메모리 개념도 ]

   프로그램 A가 보는 세계          실제 물리 메모리 (RAM)
  ┌──────────────────┐          ┌──────────────────────┐
  │ 가상 주소 0x0000 │ ───┐     │ ... 다른 프로그램 ... │
  │ 가상 주소 0x1000 │ ──┐│     ├──────────────────────┤
  │ 가상 주소 0x2000 │ ─┐││     │ 물리 주소 0x5000     │ ←─┐
  │       ...        │  │││     ├──────────────────────┤   │
  └──────────────────┘  │││     │ 물리 주소 0x8000     │ ←┐│
                        │││     ├──────────────────────┤  ││
                        ││└────→│ 물리 주소 0xA000     │  ││
                        │└─────→│ 물리 주소 0xF000     │  ││
                        └──────→│ 물리 주소 0x3000     │  ││
                                └──────────────────────┘  ││
                                                          ││
   프로그램 B가 보는 세계                                  ││
  ┌──────────────────┐                                    ││
  │ 가상 주소 0x0000 │ ───────────────────────────────────┘│
  │ 가상 주소 0x1000 │ ────────────────────────────────────┘
  └──────────────────┘
```

핵심: 프로그램은 자신이 연속된 메모리를 쓰고 있다고 "착각"하지만, 실제 물리 메모리에서는 **흩어져 있어도 괜찮다**.

### 2.2 페이지와 프레임

OS는 메모리를 **고정 크기의 작은 단위**로 쪼갠다.

| 용어 | 설명 |
|------|------|
| **페이지(Page)** | 가상 메모리를 고정 크기(보통 4KB)로 나눈 단위 |
| **프레임(Frame)** | 물리 메모리를 같은 크기로 나눈 단위 |
| **페이지 테이블(Page Table)** | 페이지 → 프레임 매핑 정보를 담은 표 |

```
[ 페이지 → 프레임 매핑 ]

  가상 메모리 (페이지)              페이지 테이블            물리 메모리 (프레임)
 ┌────────────────┐         ┌──────┬──────────┐        ┌────────────────┐
 │ 페이지 0       │────────→│  P0  │ 프레임 3 │───────→│ 프레임 0       │
 ├────────────────┤         ├──────┼──────────┤        ├────────────────┤
 │ 페이지 1       │────────→│  P1  │ 프레임 7 │──┐    │ 프레임 1       │
 ├────────────────┤         ├──────┼──────────┤  │    ├────────────────┤
 │ 페이지 2       │────────→│  P2  │ 프레임 1 │─┐│    │ 프레임 2       │
 ├────────────────┤         ├──────┼──────────┤ ││    ├────────────────┤
 │ 페이지 3       │────────→│  P3  │ 프레임 5 │┐││    │ 프레임 3 ← P0  │
 └────────────────┘         └──────┴──────────┘│││    ├────────────────┤
                                               │││    │ 프레임 4       │
                                               │││    ├────────────────┤
                                               ││└───→│ 프레임 5 ← P3  │
                                               ││     ├────────────────┤
                                               ││     │ 프레임 6       │
                                               │└────→├────────────────┤
                                               │      │ 프레임 7 ← P1  │
                                               └─────→├────────────────┤
                                                      │ 프레임 1 ← P2  │
                                                      └────────────────┘
```

### 2.3 페이징이 단편화를 해결하는 방법

- **외부 단편화 해소**: 연속된 공간이 필요 없다. 빈 프레임이 어디에 흩어져 있든 페이지 테이블로 매핑하면 된다.
- **내부 단편화 최소화**: 고정 크기 블록이므로 마지막 페이지에서만 약간의 낭비가 발생한다. 페이지 크기가 4KB라면 최대 낭비는 4KB 미만이다.

### 2.4 스왑(Swap)

물리 메모리가 부족할 때, OS는 당장 사용하지 않는 페이지를 **디스크(SSD/HDD)**로 내보낸다. 이것을 **스왑 아웃(Swap Out)**이라 한다. 나중에 다시 필요하면 **스왑 인(Swap In)**으로 물리 메모리에 되돌린다.

```
[ 스왑 메커니즘 ]

  물리 메모리 (RAM)                      디스크 (Swap 영역)
 ┌────────────────┐                    ┌────────────────┐
 │ 프레임 A (활성)│                    │                │
 ├────────────────┤   Swap Out         │                │
 │ 프레임 B (비활성)│ =================> │ 프레임 B 사본  │
 ├────────────────┤                    │                │
 │ 프레임 C (활성)│     Swap In        │                │
 ├────────────────┤ <================= │ 프레임 D 사본  │
 │ (새 프레임 D)  │                    │                │
 └────────────────┘                    └────────────────┘
```

### 2.5 Copy-on-Write (CoW)

여러 프로세스가 동일한 데이터를 읽기만 한다면, 같은 물리 프레임을 **공유**한다. 어떤 프로세스가 데이터를 **수정하려고 할 때만** 해당 프레임을 복사해서 별도의 프레임을 만들어 준다. 이로써 불필요한 메모리 복사를 줄인다.

```
[ Copy-on-Write ]

  (1) 초기 상태: 공유                 (2) 쓰기 발생 시: 복사

  프로세스 A ──┐                      프로세스 A ──→ [프레임 X']  (복사본)
               ├──→ [프레임 X]                            ↑ 수정
  프로세스 B ──┘                      프로세스 B ──→ [프레임 X]   (원본 유지)
```

## 3. LLM 서빙의 KV Cache 문제

### 3.1 KV Cache란?

Transformer 기반 LLM은 텍스트를 생성할 때 **자기회귀(autoregressive)** 방식을 사용한다. 즉, 토큰을 하나씩 순차적으로 생성하며, 이전에 생성된 모든 토큰에 대해 Attention 연산을 수행해야 한다.

이때 매번 이전 토큰들의 Key, Value 텐서를 다시 계산하면 엄청난 낭비이므로, **한 번 계산한 Key와 Value를 캐시(Cache)**에 저장해 둔다. 이것이 **KV Cache**다.

```
[ KV Cache의 역할 ]

  토큰 생성 과정:

  Step 1: "오늘"        → K₁, V₁ 계산 → KV Cache에 저장
  Step 2: "오늘 날씨"    → K₂, V₂ 계산 → KV Cache에 저장, K₁V₁ 재사용
  Step 3: "오늘 날씨 좋다" → K₃, V₃ 계산 → KV Cache에 저장, K₁V₁K₂V₂ 재사용
  ...
  Step N: N번째 토큰    → Kₙ, Vₙ 계산 → KV Cache에 저장, 이전 것 모두 재사용
```

### 3.2 KV Cache의 메모리 크기

KV Cache는 모델이 클수록, 시퀀스가 길수록 **기하급수적으로 커진다**.

단일 토큰의 KV Cache 크기 계산:

```
KV Cache (1 토큰) = 2 × num_layers × num_heads × head_dim × dtype_size

예시 — LLaMA-13B:
  = 2 × 40 × 40 × 128 × 2 bytes (FP16)
  = 819,200 bytes ≈ 800 KB / 토큰

최대 시퀀스 길이 2048 토큰 기준:
  = 800 KB × 2048 = 1.6 GB / 요청
```

A100 GPU (80GB) 기준으로, 모델 파라미터가 약 26GB를 차지하면 KV Cache에 쓸 수 있는 메모리는 약 54GB다. 이론적으로 약 33개의 요청만 동시 처리 가능하다.

### 3.3 기존 시스템의 메모리 낭비

기존 LLM 서빙 시스템(FasterTransformer, Orca 등)은 각 요청의 KV Cache를 **연속된 메모리 공간**에 할당했다. 이 방식에는 심각한 문제가 있다.

```
[ 기존 시스템의 KV Cache 메모리 할당 — 3가지 낭비 ]

  GPU 메모리 (KV Cache 영역):

  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │  요청 A: max_len=2048로 예약                                │
  │  ┌────────────┬─────────────────────────────────┐           │
  │  │ 실제 사용  │         예약만 된 공간           │ ← ① 내부  │
  │  │ (312 토큰) │        (1736 토큰 분)           │   단편화   │
  │  │   15%      │           85% 낭비!             │           │
  │  └────────────┴─────────────────────────────────┘           │
  │                                                             │
  │  요청 B: 이미 완료되어 해제됨                               │
  │  ┌─────────────────────────────────────────────┐            │
  │  │              빈 공간 (해제됨)               │            │
  │  └─────────────────────────────────────────────┘            │
  │                                                             │
  │  요청 C: max_len=2048로 예약                                │
  │  ┌──────────────────┬───────────────────────────┐           │
  │  │ 실제 사용        │      예약만 된 공간       │ ← ① 내부  │
  │  │ (1024 토큰)      │     (1024 토큰 분)       │   단편화   │
  │  └──────────────────┴───────────────────────────┘           │
  │                                                             │
  │  ② 외부 단편화: 요청 B 자리에 새 요청 D가 들어가기엔       │
  │     크기가 안 맞을 수 있음                                  │
  │                                                             │
  │  ③ 예약 낭비: 실제 생성 길이를 알 수 없어 최대 길이로 예약  │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘
```

논문에 따르면, 기존 시스템은 할당된 KV Cache 메모리의 **60 ~ 80%가 낭비**된다. 이 낭비는 세 가지 원인에서 발생한다:

| 낭비 유형 | 원인 | 영향 |
|-----------|------|------|
| **예약 낭비 (Reservation)** | 출력 길이를 예측할 수 없어 최대 길이만큼 미리 예약 | 가장 큰 비중 |
| **내부 단편화 (Internal)** | 고정 크기 슬롯에 가변 길이 시퀀스 저장 | 슬롯 내 빈 공간 |
| **외부 단편화 (External)** | 요청 완료 후 해제된 공간이 파편화 | 새 요청 할당 불가 |

40GB KV Cache 용량 기준, 실효 메모리는 **8 ~ 16GB**에 불과하다. 이는 동시 처리 가능한 배치 크기를 심각하게 제한하고, GPU 활용률을 떨어뜨린다.

## 4. PagedAttention의 핵심 아이디어

### 4.1 OS 페이징에서 영감을 받다

PagedAttention의 핵심 통찰은 간단하다:

> **KV Cache를 OS의 가상 메모리처럼 관리하자.**

| OS 개념 | PagedAttention 대응 |
|---------|---------------------|
| 프로세스 (Process) | 시퀀스 (Sequence / 요청) |
| 바이트 (Byte) | 토큰 (Token) |
| 페이지 (Page) | KV 블록 (KV Block) |
| 페이지 테이블 (Page Table) | 블록 테이블 (Block Table) |
| 가상 주소 (Virtual Address) | 논리 블록 번호 (Logical Block Number) |
| 물리 주소 (Physical Address) | 물리 블록 번호 (Physical Block Number) |
| 스왑 (Swap) | GPU ↔ CPU 메모리 스왑 |
| Copy-on-Write | KV 블록 Copy-on-Write |

### 4.2 블록 기반 KV Cache 관리

기존 시스템이 시퀀스 전체의 KV Cache를 하나의 연속된 메모리에 저장했다면, PagedAttention은 이를 **고정 크기 블록**으로 나눈다.

```
[ 기존 방식 vs PagedAttention ]

  ── 기존 방식: 연속 할당 ──

  시퀀스 "오늘 날씨 참 좋다"  (4 토큰)
  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
  │  K₁  │  K₂  │  K₃  │  K₄  │      │      │      │      │  ← 8칸 예약, 4칸만 사용
  │  V₁  │  V₂  │  V₃  │  V₄  │ 빈칸 │ 빈칸 │ 빈칸 │ 빈칸 │    50% 낭비
  └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
          연속된 메모리 공간 필요!

  ── PagedAttention: 블록 기반 ──

  시퀀스 "오늘 날씨 참 좋다"  (4 토큰, 블록 크기=2)

  논리 블록 0        논리 블록 1
  ┌──────┬──────┐   ┌──────┬──────┐
  │  K₁  │  K₂  │   │  K₃  │  K₄  │
  │  V₁  │  V₂  │   │  V₃  │  V₄  │
  └──────┴──────┘   └──────┴──────┘
       │                  │
       │ 블록 테이블       │
       ↓                  ↓
  물리 블록 7         물리 블록 2        ← 비연속 저장 가능!
  (GPU 메모리         (GPU 메모리
   어딘가)             다른 곳)
```

### 4.3 On-Demand 할당

PagedAttention은 KV Cache를 **미리 최대 길이만큼 예약하지 않는다**. 새 토큰이 생성될 때마다 필요한 만큼만 블록을 할당한다.

```
[ On-Demand 블록 할당 과정 ]

  Step 1: "오늘" (1 토큰)
  ┌──────┬──────┐
  │  K₁  │      │  ← 블록 1개 할당, 1칸 사용
  └──────┴──────┘

  Step 2: "오늘 날씨" (2 토큰)
  ┌──────┬──────┐
  │  K₁  │  K₂  │  ← 기존 블록에 추가, 블록 가득 참
  └──────┴──────┘

  Step 3: "오늘 날씨 참" (3 토큰)
  ┌──────┬──────┐  ┌──────┬──────┐
  │  K₁  │  K₂  │  │  K₃  │      │  ← 새 블록 1개 추가 할당
  └──────┴──────┘  └──────┴──────┘

  Step 4: "오늘 날씨 참 좋다" (4 토큰)
  ┌──────┬──────┐  ┌──────┬──────┐
  │  K₁  │  K₂  │  │  K₃  │  K₄  │  ← 기존 블록에 추가
  └──────┴──────┘  └──────┴──────┘

  → 낭비: 마지막 블록의 빈 슬롯만 (이 경우 0!)
```

### 4.4 단편화 해결 요약

```
┌──────────────────┬───────────────────────┬─────────────────────────┐
│   낭비 유형      │   기존 시스템         │   PagedAttention        │
├──────────────────┼───────────────────────┼─────────────────────────┤
│ 예약 낭비        │ 최대 길이 전체 예약   │ 필요할 때만 블록 할당   │

## 5. vLLM의 구현 세부사항

vLLM은 PagedAttention을 실제 시스템으로 구현한 고성능 LLM 서빙 엔진이다.

### 5.1 블록 크기 (Block Size)

vLLM의 기본 블록 크기는 **16 토큰**이다.

```
1 블록 = 16 토큰의 Key + Value 텐서

블록 크기 계산 예시 (LLaMA-13B, FP16):
  1 블록 = 16 × 2(K,V) × 40(layers) × 40(heads) × 128(head_dim) × 2(bytes)
         = 16 × 819,200 bytes
         ≈ 12.5 MB
```

블록 크기 선택에는 트레이드오프가 있다:

```
┌────────────────┬────────────────────────┬────────────────────────┐
│                │ 블록 크기 ↑ (크게)     │ 블록 크기 ↓ (작게)     │
├────────────────┼────────────────────────┼────────────────────────┤
│ GPU 연산 효율  │ ✅ 높음 (큰 행렬 연산) │ ❌ 낮음 (오버헤드 증가)│
│ 내부 단편화    │ ❌ 증가 (빈 슬롯 多)   │ ✅ 감소               │
│ 관리 오버헤드  │ ✅ 블록 수 적음        │ ❌ 블록 수 많음        │
├────────────────┼────────────────────────┼────────────────────────┤
│ 기본값: 16     │    균형점 (sweet spot)                          │
└────────────────┴─────────────────────────────────────────────────┘
```

### 5.2 블록 테이블 (Block Table)

블록 테이블은 각 시퀀스의 **논리 블록 번호 → 물리 블록 번호** 매핑을 저장하는 자료구조다.

```
[ 블록 테이블 구조 ]

  시퀀스 A (총 50 토큰, 블록 크기=16)
  ┌─────────────┬────────────┬───────────┐
  │  논리 블록  │  물리 블록  │  사용 슬롯 │
  ├─────────────┼────────────┼───────────┤
  │      0      │     7      │   16/16   │
  │      1      │     3      │   16/16   │
  │      2      │    12      │   16/16   │
  │      3      │     1      │    2/16   │  ← 마지막 블록 (14칸 빈 공간)
  └─────────────┴────────────┴───────────┘

  시퀀스 B (총 30 토큰)
  ┌─────────────┬────────────┬───────────┐
  │  논리 블록  │  물리 블록  │  사용 슬롯 │
  ├─────────────┼────────────┼───────────┤
  │      0      │     5      │   16/16   │
  │      1      │     9      │   14/16   │  ← 마지막 블록
  └─────────────┴────────────┴───────────┘

  GPU 물리 메모리 블록 배치:
  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬─ ...
  │ Bl 0 │ Bl 1 │ Bl 2 │ Bl 3 │ Bl 4 │ Bl 5 │ Bl 6 │ Bl 7 │
  │(free)│ A[3] │(free)│ A[1] │(free)│ B[0] │(free)│ A[0] │
  └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴─ ...
  ↑ 물리 블록들은 비연속적으로 흩어져 있지만, 블록 테이블을 통해 논리적으로 연속
```

### 5.3 스왑 메커니즘 (GPU ↔ CPU)

GPU 메모리가 부족해지면 vLLM은 **선점(Preemption)** 정책을 통해 일부 시퀀스의 KV Cache를 CPU 메모리로 옮긴다.

#### All-or-Nothing 방출 정책

vLLM은 하나의 시퀀스의 모든 블록을 한꺼번에 내보낸다 (일부만 내보내지 않음). 선점된 시퀀스는 두 가지 방법 중 하나로 복구된다:

```
[ 선점 및 복구 전략 ]

  방법 1: Swapping (스왑)
  ┌───────────────────┐          ┌───────────────────┐
  │    GPU 메모리     │  Swap    │    CPU 메모리     │
  │                   │  Out     │                   │
  │  시퀀스 A 블록들  │ =======> │  시퀀스 A 블록들  │
  │  [Bl3][Bl7][Bl1]  │          │  [Bl3][Bl7][Bl1]  │
  │                   │  Swap    │                   │
  │  (빈 공간 확보)   │ <======= │  (나중에 복구)    │
  │                   │  In      │                   │
  └───────────────────┘          └───────────────────┘

  방법 2: Recomputation (재계산)
  ┌───────────────────┐
  │    GPU 메모리     │
  │                   │
  │  시퀀스 A 블록들  │ ──→ 해제 (삭제)
  │  [Bl3][Bl7][Bl1]  │
  │                   │     나중에 다시 필요하면
  │  (빈 공간 확보)   │     프롬프트부터 다시 계산
  └───────────────────┘
```

- **Swapping**: 블록을 CPU RAM으로 옮기고, 나중에 다시 GPU로 가져옴. 메모리 대역폭이 병목.
- **Recomputation**: 블록을 버리고 나중에 다시 계산. 계산 비용이 발생하지만, CPU 메모리가 부족할 때 유용.

#### 스왑 우선순위

vLLM은 **FCFS(First-Come, First-Served)** 방식으로 선점 대상을 결정한다. 가장 나중에 도착한 시퀀스부터 선점하여, 먼저 도착한 요청이 우선 처리된다.

### 5.4 Copy-on-Write (CoW)

PagedAttention의 블록 기반 구조는 **메모리 공유**를 자연스럽게 지원한다. 이는 특히 **Parallel Sampling**과 **Beam Search**에서 큰 이점을 가진다.

#### Parallel Sampling 예시

하나의 프롬프트에서 여러 개의 응답을 동시에 생성하는 경우:

```
[ Copy-on-Write로 메모리 공유 — Parallel Sampling ]

  프롬프트: "인공지능의 미래는"  →  공통 KV Cache (프롬프트 부분)

  ┌─────────────────────────────────────────────────┐
  │         물리 블록 5: "인공지능의"                │
  │         물리 블록 9: "미래는"                    │
  │              ref_count = 3                       │
  └───────┬─────────────┬─────────────┬─────────────┘
          │             │             │
    응답 1의 블록 테이블  응답 2        응답 3
    [5, 9, 11]         [5, 9, 14]    [5, 9, 20]
          │                │             │
          ↓                ↓             ↓
    "밝습니다"         "어둡습니다"   "불확실합니다"
    (물리 블록 11)     (물리 블록 14) (물리 블록 20)
```

프롬프트 부분의 KV Cache(블록 5, 9)는 세 응답이 **공유**한다. 각 응답이 서로 다른 토큰을 생성하면, 그 부분만 새로운 물리 블록에 저장된다.

#### CoW가 필요한 순간

공유 블록의 마지막 슬롯에 다른 토큰을 써야 할 때:

```
[ Copy-on-Write 발동 조건 ]

  상황: 블록 9에 아직 빈 슬롯 1개가 남아있고,
        응답 1이 여기에 토큰을 쓰려 한다.
        그런데 블록 9는 ref_count = 3 (공유 중)

  (1) 블록 9를 복사하여 새 블록 15 생성
  (2) 응답 1의 블록 테이블에서 블록 9 → 블록 15로 변경
  (3) 블록 15에 새 토큰 기록
  (4) 블록 9의 ref_count = 2로 감소

  Before:                              After:
  응답1 → [5, 9, ...]                  응답1 → [5, 15, 11]
  응답2 → [5, 9, ...]                  응답2 → [5,  9, 14]
  응답3 → [5, 9, ...]                  응답3 → [5,  9, 20]
               ↑                                     ↑
          ref_count=3                            ref_count=2
                                         블록 15는 블록 9의 복사본
                                         + 응답1의 새 토큰
```

#### CoW의 메모리 절감 효과

Parallel Sampling에서 CoW를 사용하면 프롬프트 부분의 KV Cache를 복제하지 않아도 되므로, 메모리 사용량을 최대 **55%까지 절감**할 수 있다.

## 6. 성능 개선 결과

### 6.1 처리량 (Throughput) 향상

PagedAttention을 적용한 vLLM은 기존 최고 성능 시스템 대비 획기적인 처리량 개선을 달성했다.

```
[ 처리량 비교 — 기존 시스템 대비 ]

  ┌────────────────────────────────────────────────────────┐

  │                                                        │
  │  HuggingFace         ▓  1x                            │
  │  Transformers                                          │
  │                                                        │
  │  FasterTransformer   ▓▓▓▓▓▓  ~6x                     │
  │                                                        │
  │  Orca                ▓▓▓▓▓▓▓  ~7x                    │
  │                                                        │
  │  vLLM                ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  24x  │
  │  (PagedAttention)                                      │
  │                                                        │
  └────────────────────────────────────────────────────────┘

주요 성능 수치:

| 비교 대상 | 처리량 향상 | 비고 |
|-----------|------------|------|

| vs. FasterTransformer | **2 ~ 4배** | 당시 최고 성능 시스템 대비 |

### 6.3 Parallel Sampling에서의 개선

Copy-on-Write 메모리 공유를 통한 개선:

| 메트릭 | 기존 시스템 | vLLM | 개선 |
|--------|-----------|------|------|
| 메모리 사용량 | 100% (모두 복제) | 45% | **55% 절감** |
| 처리량 | 1x | **2.2x** | 2.2배 향상 |

### 6.4 Beam Search에서의 개선

Beam Search는 여러 후보 시퀀스가 공통 접두어(prefix)를 공유하는 특성이 있어, CoW의 효과가 극대화된다.

### 6.5 후속 버전의 추가 최적화

vLLM v0.6.0 (2024년 9월)에서는 추가적인 최적화가 적용되었다:

| 최적화 기법 | 효과 |
|-------------|------|
| Multi-step scheduling | GPU 유휴 시간 감소, 처리량 1.8 ~ 2.7배 추가 향상 |
| Object cache | Python 오브젝트 캐싱으로 24% 처리량 향상 |
| Torch compile 통합 | 커널 최적화 |

### 6.6 실용적 의미

PagedAttention이 가져온 효율 개선은 단순한 수치 이상의 의미가 있다:

```
[ 비용 절감 효과 — 동일 워크로드 기준 ]

  기존 시스템:  GPU 4대 필요  →  월 비용: $$$$$$$$$$$$$$$
  vLLM:        GPU 1대 충분  →  월 비용: $$$$

  2 ~ 4배 처리량 향상 = GPU 2 ~ 4대 추가 구매와 동등한 효과
  → 하드웨어 투자 없이 소프트웨어만으로 달성
```
